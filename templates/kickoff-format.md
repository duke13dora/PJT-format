# プロジェクトキックオフ フォーマット

> **用途**: プロジェクト開始時の合意形成資料。MDで中身を擦り合わせ → 合意後にPPTXスライド化。
> **運用フロー**: ① このフォーマットに沿ってMDで記述 → ② 関係者と認識合わせ → ③ 合意後にPPTX資料化（※PPTX化の手引きは別途作成予定） → ④ 合意後は変更しない。変更が必要な場合は定例で合意の上、版を上げる
> **バージョン**: v1.0（2026-03-19）

---

## フォーマット構成

```
1. プロジェクト概要
   1.1 背景（なぜ今やるのか）
   1.2 目的・スコープ
   1.3 体制・役割分担

2. 弊社理解（現状認識とゴール）
   2.1 現状認識（現時点で把握している状況）
   2.2 ゴール（目指す姿）

3. 論点
   - プロジェクトを進める上で判断が必要な事項

4. アプローチ・実施事項・アウトプット
   4.1 フェーズと実施事項
   4.2 IPO整理（実施事項別のインプット・プロセス・アウトプット）

5. マイルストーン

6. リスクと前提条件
   6.1 前提条件（計画の成立要件）
   6.2 リスク（発生可能性のある問題と対応方針）
```

## 各セクションの記述ガイド

### 1. プロジェクト概要

#### 1.1 背景（なぜ今やるのか）

**目的**: プロジェクトの動機を共有する。途中参加メンバーや経営報告時にも使い回せるように。

**書き方**:
- 「このままだとどうなるか（Pain）」→「変わるとどうなるか（Gain）」のストーリー
- 事実ベースで書く（曖昧な表現は禁止）
- 3〜5行程度に収める

#### 1.2 目的・スコープ

**目的**: 何を達成するPJか、何がスコープ内/外かを明確にする。

**書き方**:
- 目的: 1文で端的に
- スコープ内: 箇条書き
- スコープ外: 明示的に「やらないこと」を書く（スコープクリープ防止）

#### 1.3 体制・役割分担

**目的**: 誰が何を担当するかの合意。

**書き方**:
- テーブル形式（組織 / 氏名 / 役割 / 備考）
- 意思決定者・実行者・レビュワーの区別を明確に

---

### 2. 弊社理解（現状認識とゴール）

#### 2.1 現状認識

**目的**: 「我々はこう理解していますが合っていますか？」の認識合わせ。

**注意**:
- これは**分析結果ではない**。PJ開始時点での弊社の理解。
- 詳細なAs-Is分析はPJ成果物として別途作成する。
- 事実と推論を混同しない。確信度が低いものは「〜と推察」等を付記。

#### 2.2 ゴール

**目的**: このPJで目指す到達点の合意。

**書き方**:
- 定性ゴール: 「〜が実現されている状態」
- 定量KPI（設定可能であれば）: 「〜率が○%以上」
- ※KPIはPJ進行中に精緻化する前提で、初期仮説でOK

---

### 3. 論点

**目的**: PJを進める上で判断が必要な事項を先出しする。

**書き方**:
- 大論点（3〜5個程度）に分類
- 各論点の下に小論点を箇条書き
- 「答え」は書かない。「判断が必要であること」を共有する場。
- ※ギャップ分析はPJ成果物。キックオフ時点ではギャップが明確になっていないため、「論点」として提示。

---

### 4. アプローチ・実施事項・アウトプット

#### 4.1 フェーズと実施事項

**目的**: どういう順序で何をやるかの合意。

**書き方**:
- フェーズ分け（例: 現状把握 → 設計 → 実装 → 展開）
- 各フェーズの実施事項をテーブルで整理
- テーブル列: # / フェーズ / 実施事項 / 済/未 / 備考 / 対応論点

#### 4.2 IPO整理（実施事項別のインプット・プロセス・アウトプット）

**目的**: 各実施事項のインプット・プロセス・アウトプットを明確にする。「このPJで何を納品するのか」の合意。スコープクリープ防止の最重要セクション。

**書き方**:
- テーブル列: # / 実施事項 / インプット / プロセス / アウトプット
- 実施事項ごとにIPOを整理する（フェーズ別ではない）
- アウトプットには形式（MD/PPTX/Excel等）を併記
- 実施事項 = Backlog親課題、アウトプット = Backlog子課題として運用する前提で設計
- 「成果物としてここまでは出す、ここからは出さない」を明確に

**設計思想**:
- 子課題（アウトプット）を全て完了すれば、親課題（実施事項）がクローズする構造
- インプットの依存関係により、実施事項間の順序が自然に決まる

---

### 5. マイルストーン

**目的**: いつまでに何が完了するかの合意。

**書き方**:
- 月次マイルストーンのテーブル（時期 / マイルストーン）
- **マイルストーンの擦り合わせのみ**。詳細スケジュールは別途PPTX等で管理し、キックオフMDからはリンク参照する（二重更新の回避）
- 留意事項（制約条件、依存関係）を併記

---

### 6. リスクと前提条件

#### 6.1 前提条件

**目的**: 「これが成り立たないと計画が崩れる」条件の合意。

**書き方**:
- 箇条書きで明示
- 例: 「○○環境が利用可能であること」「○○のリソースが確保されていること」

#### 6.2 リスク

**目的**: 「起きる可能性があり、起きたら影響がある」事項の共有。

**書き方**:
- テーブル列: # / リスク / 影響度 / 発生可能性 / 対応方針
- 既知のリスクを承知の上で進めるという合意を取る

## 別管理にするもの（キックオフMDには含めない）

| 資料 | 管理場所 | 備考 |
|------|---------|------|
| ファクトベース（事実一覧） | FACTS.md + アジェンダシート | 出典付きの確認済み事実 |
| 詳細スケジュール | 全体スケジュールPPTX | キックオフMDからはリンク参照のみ |
| 未確定事項・次のアクション | Backlogチケット + 定例 | 流動的な情報のため別管理 |
| 詳細As-Is分析 | PJ成果物として別途 | PJ進行中に作成 |
| 詳細To-Beフロー | PJ成果物として別途 | PJ進行中に作成 |
| 専門家ディスカッション | 社内検討ログ | 意思決定の根拠として保持 |
| 既存資料一覧 | 事前ヒアリング準備MD等 | ファクトベースの出典として参照 |
