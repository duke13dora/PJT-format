## API操作の基本ルール

### 日本語テキスト送信

- **Node.js fetch/https.request を使用**（curlはWindows UTF-8文字化け）
- **大きなペイロード（10K文字超）**: `fetch` API推奨。`https.request` + `req.write(string)` は大きなペイロードでU+FFFD文字化けが発生する
  ```javascript
  // 安全な方法
  fetch(url, { method: 'PATCH', body: 'description=' + encodeURIComponent(text) });
  ```
- **投稿・編集後の必須確認**: レスポンスのcontentを確認してから完了報告すること

### 環境変数

| 変数名 | 内容 | 注意 |
|--------|------|------|
| `BACKLOG_DOMAIN` | ホスト名のみ（例: `xxx.backlog.com`） | ❌ `BACKLOG_SPACE_URL` は誤り |
| `BACKLOG_API_KEY` | APIキー | `.mcp.json` の `env` で管理 |

## メンション・通知

### テキスト「○○様」では通知されない

Backlogのコメント本文に「○○様」と書いても、相手には**通知されない**。必ず `notifiedUserId[]` パラメータを使うこと。

```javascript
// MCP経由でも同様にnotifiedUserIdを指定
{
  "content": "ご確認をお願いいたします。",
  "notifiedUserId": [159135, 160089]  // 先方のユーザーID
}
```

### 運用ルール

- **ユーザーIDはプロジェクトMEMORY.mdに記録しておく**
- **「先方にメンション」= 先方メンバー全員に通知**
- **チケット説明前に担当者を振らない**: 先方に未説明のチケットを急に振ると混乱を招く

## コメント書式（プレーンテキスト厳守）

### マークダウン禁止

Backlogコメントでは `**太字**` 等のマークダウン記法は**そのまま文字として表示される**。AI感が出るため使用禁止。

### 構造記号

| 記号 | 用途 | 例 |
|------|------|-----|
| ■ | 大項目の見出し | `■ 対応内容` |
| ※ | 補足・経緯・意図 | `※既存の設定を活かす方針で対応しました` |
| → | アクション・結果 | `→資料を格納しました` |
| - | 箇条書き | `- 追加コストが不要` |

### コメント構成パターン

```
冒頭の謝意（1行）

■ 項目1
内容説明
※補足・経緯

■ 項目2
内容説明
→結果・アクション

- 箇条書き点1
- 箇条書き点2
```

### 冒頭謝意の使い分け

| 状況 | 表現 |
|------|------|
| 指摘を受けた | 「ご指摘ありがとうございます。」 |
| 意見・情報提供 | 「ご意見ありがとうございます。」 |
| ミスの指摘 | 「〜ができておらず失礼いたしました。今後は〜するようにいたします。」 |
| 確認依頼 | 「ご確認をお願いいたします。」 |

## チケット起票ルール

### 必須設定項目

- **カテゴリー**: 必ず設定（未設定だとボードに表示されない）
- **マイルストーン**: 必ず設定（該当期のマイルストーンを選択）
- **起票前に既存チケット検索**: `get_issues` でキーワード検索し、重複起票を防ぐ

### ボール管理

- **先方ボール時**: タイトルに「→資料確認」「→修正確認」等を追記 + 担当者を先方に変更
- **自社ボール時**: タイトルからボール表記を除去 + 担当者を自社メンバーに戻す

## 資料共有フロー

### 先方に資料を共有する手順

1. **概要欄を更新**:
   - Backlog共有ファイルURL
   - 「確認を容易にするために上記資料の内容を下記に添付します。」
   - `---`（区切り線）
   - 資料全文ベタ張り

2. **コメント投稿**（プレーンテキスト）:
   - 資料名 + パスURL + 概要欄にも全文掲載の旨 + 確認依頼

3. **notifiedUserId**: 先方メンバー全員を指定

4. **共有ファイルのアップロード**: ユーザーにリマインド（後述の制約参照）

### 共有ファイルの制約

- **アップロードAPIは存在しない**: `POST /projects/:key/files` → 400。**Web UIから手動アップロードのみ**
- **URL形式**: `https://{domain}/file/{PROJECT_KEY}/{URLエンコード済みパス}/{URLエンコード済みファイル名}`
- ❌ `ViewSharedFile.action?sharedFileId=...` → 404
- ❌ `/projects/{KEY}/files` → 404

### 資料作成の順序（厳守）

1. 資料を作成（Notion / MD等）
2. **ユーザーが確認・認識合わせ**
3. 合意後にBacklogチケットの説明欄を更新

**② の合意前にチケット更新をしない**（手戻りが大きい）

## 親子チケット運用

### 親チケットの概要欄

**静的に保つ**: 資料パス + 「詳細は子課題を参照してください。」のみ。概要・子課題一覧等の動的情報は書かない（毎回更新が必要になるため）。

### 子チケットの概要欄

資料パス + 全文ベタ張りフォーマットを適用。更新時はコメント「概要欄を更新しました。」を投稿。

### 制約

- 子を持つチケットは親にできない（先に子を移動してから親子関係を変更）
- ガントチャートの子課題折りたたみは標準機能なし
- ボードの子課題非表示は標準機能なし（カテゴリーフィルターで対処）

---

## 既知の制約

| 制約 | 対処法 |
|------|--------|
| 共有ファイルアップロードAPI不存在 | Web UIで手動アップロード |
| ゲストユーザーはチケット削除不可 | `old_` プレフィックス + 完了で対処 |
| curlで日本語文字化け | Node.js fetch/https.request を使用 |
| Backlog Wiki | Markdownモードで記述（`#`/`##`/`###`を使用） |
| テキスト宛名は通知されない | `notifiedUserId[]` パラメータ必須 |
