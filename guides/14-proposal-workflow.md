# 提案書の作り方

> 提案書を MD → すり合わせ → PPTX の3フェーズで作成するフロー。PJT-SEで確立した手法。最終報告書（08参照）と同じパイプラインで、セクション構成と設計原則が異なる。

---

## 全体フロー

```
⓪ 情報収集（Phase 0）
   ├─ 0-1. 案件ヒアリング（シナリオ判定）
   ├─ 0-2. リサーチ計画策定
   ├─ 0-3. Gemini Deep Research実行（ユーザー手動）
   └─ 0-4. 情報パッケージ作成 → [案件名]_research.md
       │
       ▼
① 提案書MDの作成
   └─ templates/proposal-format.md + 情報パッケージをインプットに記述
       │
       ▼
② ユーザーとすり合わせ
   └─ MDベースで認識合わせ。中身の合意が取れてからPPTX化に進む
       │
       ▼
③ PPTX生成（FLUXフォーマット流し込み）
   └─ scripts/build-proposal-example.mjs をカスタマイズして実行
       │
       ▼
④ レビュー・微修正
   └─ PowerPointで開いて位置ずれ・重なりを手動修正
```

## フェーズ0: 情報収集

> 提案書MDを書き始める**前に**、必要な情報を構造的に収集するフェーズ。提案の精度は入力情報の質で決まる。

### 0-1. 案件ヒアリング（シナリオ判定 + 方向性すり合わせ）

Claudeがユーザーに以下を**一括で**質問する（逐次質問禁止）:

| 質問項目 | 目的 |
|---------|------|
| クライアント名・業種 | 基本情報 |
| 商談経緯（紹介/問い合わせ/ノードアノック） | シナリオ判定 |
| 既存の接点・商談履歴の有無 | 情報の充足度判定 |
| 提案の目的（新規/アップセル/クロスセル） | 提案の方向性 |
| 提案先の意思決定者・キーパーソン | 読者の特定 |
| 予算感・時期の制約 | スコープ制約 |
| 手元にある資料（議事録/IR/過去提案等） | 既存情報の棚卸し |

回答をもとに**シナリオを判定**し、リサーチの深さを決定する:

| シナリオ | 判定条件 | リサーチ重点 | 目安所要時間 |
|---------|---------|------------|------------|
| Warm（既存商談あり） | 議事録あり + 具体ニーズ把握済み | 補足データのみ（市場/競合） | 1-2時間 |
| Referral（紹介） | 紹介者からの文脈あり | 企業分析 + 業界動向 | 2-3時間 |
| Cold（ノードアノック） | 接点なし、公開情報のみ | 全方位リサーチ | 3-5時間 |

シナリオ判定後、CLAUDE.mdの動的専門家ペルソナルールに従い**2名の専門家を召喚**し、提案の方向性についてディスカッションする。

### 0-2. リサーチ計画の策定

「手元にある情報」と「不足している情報」をマトリクスで整理する。不足情報は**提案書のどのセクション（02現状分析/03提案戦略/04効果見込み等）で使うか**を紐付ける。

テンプレート: `templates/research-plan.md` をコピーして使用。

**シナリオ別のリサーチ項目**:

| リサーチ項目 | Warm | Referral | Cold |
|-------------|:----:|:--------:|:----:|
| クライアント企業分析（IR/経営方針） | — | ✅ | ✅ |
| 業界動向・構造的課題 | — | ✅ | ✅ |
| 市場データ（議事録の裏付け） | ✅ | ✅ | ✅ |
| 競合/類似事例 | ✅ | ✅ | ✅ |
| 補助金・助成金 | 必要時 | 必要時 | ✅ |
| 意思決定者プロフィール | — | — | ✅ |
| 課題仮説の検証 | — | ✅ | ✅ |

### 0-3. リサーチ実行（Gemini Deep Research）

Claudeが**1テーマ1プロンプト**でGemini Deep Research用プロンプトを生成する。ユーザーがGeminiに投入し、結果をClaudeにフィードバックする。

**Geminiプロンプトの構造テンプレート**:

```
## 調査依頼: [調査項目名]

### 背景
[なぜこの調査が必要か。提案書のどのセクションで使うか。1-2文]

### 調査対象
- [具体的な対象の範囲を明記]

### 調べてほしいこと（優先順）
1. [最も重要な調査ポイント]
2. [次に重要な調査ポイント]
3. [あれば嬉しい追加情報]

### 出力フォーマット
- 数値データは出典付きで記載
- 日本語で回答
- 推定値の場合は「推定」と明記
- 情報の鮮度（何年のデータか）を明記

### 制約
- 対象地域: [日本/グローバル等]
- 対象期間: [直近3年等]
```

**プロンプト設計原則**:

1. **1プロンプト1テーマ**: 複合テーマは精度が下がる
2. **提案書セクション紐付け**: どのセクションの根拠になるか明記
3. **数値+出典を要求**: 「具体的な数値を含めて」「出典元を付記」を必ず指示
4. **推定/ファクト区別**: 推定値と確定値を区別させる
5. **日本語指定**: Geminiは英語で返す場合があるため明示

**ユーザーの作業フロー**:

1. Claudeがプロンプトを全件まとめて出力
2. ユーザーがGemini Deep Researchに1件ずつ投入
3. 結果をコピペでClaudeに戻す（または `.md`/`.docx` として `drive/` に配置）
4. Claudeが結果を読み込み、不足があれば追加プロンプトを生成

### 0-4. 情報パッケージの作成

Phase 0の成果物を `[案件名]_research.md` として構造化し、Phase 1のインプットにする。テンプレートの「情報パッケージ」セクションに結果を記入する。

主な項目:
- クライアント概要（正式名称/業種/規模/キーパーソン）
- 商談コンテキスト（経緯/顕在ニーズ/潜在ニーズ/予算/時期）
- 市場・業界データ（Geminiリサーチ結果、出典付き）
- 競合・類似事例
- 補助金・助成金情報（該当時）
- 提案仮説（根拠データ参照先付き）
- Phase 1への申し送り（ストーリーライン案/強調すべきデータ/避けるべきトピック/未解決ギャップ）

### シナリオ別フロー例

**Warm（例: オイシックス新潟）**:
```
0-1: 議事メモ.docxから過去商談3回分を読み込み → ニーズ特定済み
0-2: 市場データ補足 + 補助金調査の2項目 → リサーチ計画作成
0-3: Geminiプロンプト2本生成 → ユーザー実行 → 結果フィードバック
0-4: 情報パッケージ作成 → Phase 1へ
```

**Referral（紹介経由）**:
```
0-1: 紹介者からの情報を整理 → 仮説構築
0-2: 企業分析 + 業界動向 + 競合事例の3項目
0-3: Geminiプロンプト3本 → ユーザー実行 → 結果フィードバック
0-4: 情報パッケージ作成 → Phase 1へ
```

**Cold（ノードアノック）**:
```
0-1: クライアント名のみ → 全方位ヒアリング
0-2: 企業分析 + 業界動向 + 競合事例 + 補助金 + 意思決定者の5項目
0-3: Geminiプロンプト4-5本 → ユーザー実行 → 結果フィードバック
     → 不足あれば0-3を1回リピート
0-4: 情報パッケージ作成 → Phase 1へ
```

## フェーズ1: 提案書MDの作成

### 7セクション構成

| # | セクション | 目的 | ページ目安 |
|---|----------|------|----------|
| — | 表紙+アジェンダ | — | 2枚 |
| 01 | エグゼクティブサマリ | ここだけ読めば判断できる | 扉+1-2枚 |
| 02 | 現状分析 | ポジティブ事実→課題特定 | 扉+2-4枚 |
| 03 | 提案戦略 | 戦略の全体像＋各施策の詳細 | 扉+3-6枚 |
| 04 | 効果見込み | 成長ロードマップ・KPI目標・投資/ROI | 扉+3-5枚 |
| 05 | 実行計画 | タイムライン＋リスクと対策 | 扉+2-3枚 |
| 06 | ネクストステップ | 次のアクション・CTA | 1-2枚 |
| 07 | 参考資料 | 必要な人だけ見る詳細 | 必要に応じて |

### 最終報告書（08）との構成差分

| 項目 | 提案書（本ガイド） | 最終報告書（08） |
|------|-------------------|----------------|
| セクション数 | 7 | 5 |
| 02 | 現状分析（ポジティブ→課題） | プロジェクト概要（体制・スコープ） |
| 03 | 提案戦略（施策詳細） | 実施内容と成果 |
| 04 | 効果見込み（ROI・ロードマップ） | 課題と今後のアクション |
| 05 | 実行計画 | 参考資料 |
| 06-07 | ネクストステップ＋参考資料 | — |
| ストーリー | 課題→提案→効果（提案型） | 概要→成果→課題（報告型） |

### 設計原則

1. **ピラミッド原則**: 結論ファーストで構成。エグゼクティブサマリだけ読めば判断できる
2. **タイトルは結論を言い切る形**（疑問形・名詞止め禁止）
   - ❌「チケット売上について」→ ✅「チケット売上は動員に追いついていない」
   - ❌「施策案」→ ✅「3つのレバーで+50%の成長を実現する」
3. **リード文には具体的な数字・事実を含める**
4. **検討プロセスの記録**: 召喚した専門家・リサーチソース・主要論点を「検討プロセス」セクションに記載（スライド非掲載・社内用）
5. **読者3層を意識**: 経営層（①のみ）/ マネージャー層（①+③+④）/ 実務担当者（全ページ）

### 検討プロセスセクション

提案書MD冒頭に以下のテンプレートを記載する。PPTX化の対象外（社内用）。

```markdown
## 検討プロセス（社内用・スライド非掲載）

### 召喚した専門家
- **専門家1**: [ペルソナ名]（選定理由: [なぜこの専門家が適切か]）
- **専門家2**: [ペルソナ名]（選定理由: [なぜこの専門家が適切か]）

### リサーチソース
- [調査内容と実施日]

### ビジネス観点（ユーザー確認済み）
- [ビジネス上の目的・制約]

### 主要な論点と結論
- [論点1]: [結論]
- [論点2]: [結論]
```

### MD記法

各ページは以下の形式で記述する:

```markdown
## Page N: [ページ名]
<!-- type: [タイプ] | section: [セクション番号] -->

**タイトル**: [結論を言い切る形のタイトル]

**リード文**: [1-2文。このスライドの「So What」。数字か具体的事実を含める]

**ボディ**:
[テーブル / 箇条書き / ASCII図 / テキスト]

<!-- notes: [プレゼンター用メモ] -->

---
```

### ユーザーとの認識合わせ

- MDの段階でユーザーに内容確認を依頼
- **中身の合意が取れてからPPTX化に進む**（PPTX化後の大幅な内容変更は手戻りが大きい）
- MD上でスライド枚数・各スライドの見出し・キーメッセージを確定させる
- 質問は1回のやり取りでまとめて投げる（逐次質問は負担が大きい）

## フェーズ2: PPTX生成

### テンプレートPPTXの準備

- テンプレート: `templates/flux-presentation-template.pptx`（FLUX汎用PPTXテンプレート。5種のレイアウトサンプル付き）
- 保持する構成要素:
  - ロゴ（slideMaster内）
  - フッター
  - ヘッダーバー
  - スライドレイアウト（Layout1/7/8/95）
- **方針**: テンプレートに新規スライドを生成して追加

### 生成スクリプトの処理フロー

`scripts/build-proposal-example.mjs` をベースにカスタマイズ:

```
テンプレートPPTX (adm-zip読込)
    ↓
提案書.md をパース:
  - Page N の種別判定 (表紙/アジェンダ/扉/コンテンツ)
  - タイトル/リード/ボディの抽出
  - ボディ内のテーブル/テキストの分離
    ↓
各スライドのXMLを生成:
  - cover: buildCoverSlide()  ← 提案書固有
  - agenda: buildAgendaSlide() ← 提案書固有
  - section: buildSectionSlide()
  - content: buildContentSlide() + makeTable/makeTextBox/makeCalloutBox
    ↓
presentation.xml / .rels / [Content_Types].xml 更新
    ↓
新PPTX出力
```

### 提案書固有の関数

#### buildCoverSlide()

テンプレートの表紙スライド（slideLayout1）のプレースホルダを置換:

```javascript
function buildCoverSlide(coverTemplate, title, clientName, date) {
  let xml = coverTemplate;
  // title プレースホルダ → 提案タイトル
  // subTitle プレースホルダ → クライアント名 + 日付
  xml = xml.replace(/<p:sp>([\s\S]*?)<\/p:sp>/g, (match, content) => {
    if (content.includes('type="ctrTitle"') || content.includes('type="title"')) {
      return match.replace(/<a:t>[^<]*<\/a:t>/g, `<a:t>${escXml(title)}</a:t>`);
    }
    if (content.includes('type="subTitle"')) {
      return match.replace(/<a:t>[^<]*<\/a:t>/g, `<a:t>${escXml(clientName)} 御中\n${escXml(date)}</a:t>`);
    }
    return match;
  });
  return xml;
}
```

#### buildAgendaSlide()

アジェンダスライド（slideLayout7）の番号付きリストを `<a:p>` 単位で段落再構築:

```javascript
function buildAgendaSlide(agendaTemplate, items, pageNum) {
  // ★重要: <a:t>の逐次置換はNG → <p:txBody>ごと再構築する
  // 理由: 同一段落内の複数<a:t>をまたいで破壊するため

  // bodyプレースホルダの<p:txBody>を丸ごと差し替え
  const paragraphs = items.map(item =>
    `<a:p><a:pPr marL="342900" indent="-342900">
       <a:buAutoNum type="arabicPeriod"/>
     </a:pPr>
     <a:r><a:rPr lang="ja-JP" sz="1400" dirty="0">
       <a:latin typeface="{FONT_FACE}"/><a:ea typeface="{FONT_FACE}"/>
     </a:rPr><a:t>${escXml(item)}</a:t></a:r></a:p>`
  ).join('');
  // ... <p:txBody>全体を再構築して差し替え
}
```

**なぜ `<a:t>` 逐次置換がNGか**: 詳細は [09-pptx-xml-editing.md](09-pptx-xml-editing.md) の「番号付きリストの段落再構築」セクション参照。

### XML構造の注意点（★重要）

| 項目 | 正しい方法 | NG / 原因 |
|------|---------|---------|
| テーブルセル | `<a:tc><a:txBody><a:bodyPr/><a:lstStyle/><a:p>...</a:p></a:txBody><a:tcPr>...</a:tcPr></a:tc>` | `<a:p>`を`<a:txBody>`で囲まないとセル内テキストが表示されない |
| rId（リレーションID） | rId100+i 等、既存rIdと衝突しない番号 | テンプレートのnotesMaster(rId26), fonts(rId27+)と衝突→スライド消失 |
| graphicFrame | `<p:graphicFrame>`でテーブルをラップ。`<p:xfrm>`で位置指定 | `<a:xfrm>`ではなく`<p:xfrm>`を使用 |
| notesSlide | スライドrelsからnotesSlide参照を除去 + 孤立ファイル削除 | 共有参照＋孤立ファイル→修復ダイアログ |

共通の教訓は [09-pptx-xml-editing.md](09-pptx-xml-editing.md) に詳述。

### レイアウト・位置ルール

| 項目 | ルール | 備考 |
|------|-------|------|
| スライドサイズ | 9144000 × 5143500 EMU（16:9） | 1inch = 914400 EMU |
| ボディ開始Y | y=1.5in以上 | リードテキスト下端の下 |
| フッター領域 | y=5.0in以下に収める | ロゴ+フッターテキストがy=5.0〜5.6にある |
| テーブル後の要素Y | テーブル下端 = y + (行数+1) × rowH | 後続要素はテーブル下端+0.1in以上 |
| フォントサイズ | sz="900"=9pt, sz="1000"=10pt | OOXMLは百分の一ポイント単位 |
| テキスト色 | 全て黒（000000） | 灰色（434343等）は禁止 |

### PPTXフォーマットルール

- **テーブル行番号**: 全テーブルに#列を必ず付ける。プレーン数字（1,2,3）を使用。丸数字（①②③）禁止
- **行の交互色**: ヘッダー=`{PRIMARY_COLOR}`白文字、奇数行=白、偶数行=薄色（例: F2F7FF）
- **行高さ統一**: 単一行は全テーブルで統一（0.28"）。複数行セルのみオーバーライド
- **リード文**: 12pt、黒色、文末に句点（。）必須
- **テーブル横幅**: リード文の幅と揃える（約9.2インチ）
- **フォント**: 全てメイリオで統一
- **ボディテキスト（テーブル外）**: 10pt（sz=1000）
- **アジェンダ**: テキスト形式で表示（カード/ボックス形式禁止）。セクション名は実際のセクション扉と一致させる
- **セクションラベル**: 太字
- **見出しラベル**: 太字、`{PRIMARY_COLOR}`色
- **コールアウトボックス**: 角丸四角形 roundRect adj=5000、背景色+テキスト色を指定
  - 青系: 背景`D3E2FF` / テキスト`{PRIMARY_COLOR}` — 結論・まとめ用
  - 橙系: 背景`FFF3E0` / テキスト`ED7D31` — 期待効果・注意用

- [ ] 灰色テキストが残っていないこと

### よくあるフィードバックと対処法

| フィードバック | 対処 |
|-------------|------|
| テーブルの文字が見切れる | rowHを大きくする / fontSizeを小さくする |
| 要素が重なっている | 後続要素のy座標を下げる |
| フォントが違う | `typeface="メイリオ"`を確認 |
| 色が違う | srgbClr valを確認 |
| セクション扉が空 | buildSectionSlideのプレースホルダ置換ロジックを確認 |
| アジェンダの番号がずれる | `<a:buAutoNum type="arabicPeriod"/>`を確認 |

### 重要原則

**XML直接編集の精度限界**: レイアウト微調整、フォントの再適用、コネクタの接続点ずれ等は完璧にできない。**最終的に人間がファイルを開いて目視確認・微修正する前提で作業すること。**
