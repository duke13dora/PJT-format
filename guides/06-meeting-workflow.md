# 定例運用フロー

> 定例会議の前後で実施するタスクの手順書。PJT-YPで確立したワークフロー。

---

## 定例前ワークフロー

定例前日または当日朝に実施。

### タスク1: スケジュール赤線の移動

1. `create-schedule-pptx.mjs` の `todayWeekFloat` を書き換え
2. `node create-schedule-pptx.mjs` を実行
3. 生成された `schedule.pptx` を確認
4. `drive/50.打ち合わせ関連/` に上書きコピー

**todayWeekFloat の計算方法**:
```
weekIndex + dayOffset/7
月曜=0, 火曜=1/7, 水曜=2/7, 木曜=3/7, 金曜=5/7
```

例: 金曜日の定例で第4週 → `todayWeekFloat = 4 + 5/7`

### タスク2: アジェンダシートの更新

1. スケジュールの `rawData` を参照し、今週の赤線が被るタスクを特定
2. Backlogで対象チケットの最新状況を確認
3. すり合わせが必要なトピックを選定
4. 前回アジェンダ行を `done` 化
5. 新規アジェンダ行を追加（adm-zip XML直接編集）
6. JSZipで検証

**アジェンダの構成**:
- **Row2（standing）**: スケジュール確認 — 今週のタスク一覧を記載
- **個別トピック行**: チケットごとに1行。提案型構造:
  - ▼前提（背景・制約）
  - ▼提案（FLUX-G提案内容）
  - ▼確認事項（合意したいポイント）

**並び順**: 戦略・ガバナンス系を先、技術・作業系を後

## 定例後ワークフロー

定例終了後に実施。

```
文字起こし受領
  │
  ▼
ステップ1: 抽出（チャットで一覧提示）
  ├─ 決定事項（番号付き）
  ├─ 議事メモ（議論の要点サマリ）
  └─ ネクスト（次のアクション一覧）
  │
  ▼
ステップ2: 記録（確認不要で即実行）
  ├─ DECISIONS.md 更新（通し番号を継続）
  ├─ Excel 決定事項シート更新（adm-zip）
  ├─ Excel ネクストシート更新（adm-zip: 新規追加＋既存行の備考追記・ステータス変更）
  └─ docx 議事メモ更新（JSZip）
  → 完了報告 → ユーザー確認 → 修正あれば対応
  │
  ▼
ステップ3: ネクスト → フォーマットで一覧出力 → ユーザー確認
  → OKが出たらBacklog操作実行
```

### 文字起こし読み取りの注意事項

- **話者帰属は信頼しない**: マイク共有の場合あり。文脈から実際の話者を判断
- **日本の祝日は必ず自分で確認**: 文字起こしの「来週祝日」等を鵜呑みにしない
- **曖昧語禁止**: 具体的な数値・名称で記述

### DECISIONS.md更新のフォーマット

```
| # | 決定日 | 決定者 | 概要 | 詳細 | 備考 |
```

- 決定日: `YY/MM/DD` 形式
- 決定者: `合同` / `{会社名} {氏名}` 等
- 備考: 定例日付を記載（例: `260313定例`）

### ネクスト洗い出し — 2パターンの分岐

ネクストの管理先によってフローが異なる。プロジェクト開始時にどちらを採用するか決定し、MEMORY.mdに記録する。

#### パターンA: アジェンダシート直接管理型（★PJT-TS方式）

ネクストをExcelアジェンダシート内のネクストシートで一元管理するパターン。
Backlog/Asana等の外部プロジェクト管理ツールを使わない場合に採用。

**フロー**:
1. 文字起こしからネクストアクション（担当, 期限, 内容）を抽出
2. **そのままExcelのネクストシートに行追加**（adm-zip XML直接編集）
3. 既存行のステータス更新（done/ing/yet）も同時に実施
4. 完了報告 → ユーザー確認 → 修正あれば対応

**ネクストシートの列構成**:

| 列 | 内容 |
|----|------|
| A | 番号（通し） |
| B | 報告日 |
| C | 報告者 |
| D | サマリー |
| E | 詳細 |
| F | 備考 |
| G | 担当者 |
| H | 期限 |
| I | ステータス（done/ing/yet） |

#### パターンB: プロジェクト管理ツール型（★PJT-TD/YP方式）

ネクストをBacklog/Asana等のプロジェクト管理ツールでチケット管理するパターン。

**フロー**:
1. 文字起こしからネクストアクションを抽出
2. **チャットで以下のテーブル形式にて一覧出力**（まだ投稿しない）
3. ユーザー確認 → OKが出たらチケット操作を実行

**出力テーブルフォーマット**:

| 項目 | 内容 |
|------|------|
| 内容 | やること概要 |
| 担当者 | チケットの担当者 |
| メンション | 通知する宛先 |
| 期限 | いつまでか |
| 紐づけ先 | 既存チケットの子課題 or 新規チケット |
| 対応方法 | 子課題作成 / コメント追記 / 新規チケット / 既存チケット更新 |

**投稿用コメント文の出力**:
各チケットに対して投稿するコメント文案もテーブルと合わせて出力する。

| チケット名 | 宛先 | コメント案 |
|-----------|------|-----------|
| （具体名） | @担当者 cc @他メンバー | （投稿文面） |

ユーザーの確認を得てから一括投稿する。**確認前の投稿は厳禁**。

## 絶対厳守ルール

1. **客先に影響する操作は全て事前確認必須**: Backlogコメント投稿、担当者変更、ステータス変更
2. **チケット起票時**: カテゴリ・マイルストーンを必ず設定
3. **起票前に既存チケット検索**: 重複防止
4. **Backlogコメントはプレーンテキスト**: マークダウン記法禁止（AI感が出る）

→ Backlog運用の詳細（メンション、資料共有フロー、親子チケット等）: [guides/10-backlog-operations.md](10-backlog-operations.md)

## AI感排除原則（★PJT-TD教訓）

定例関連の全てのコミュニケーション（チケットコメント、議事メモ等）で、自動化ツールの存在を一切言及しない。

### 禁止表現

| ❌ 禁止 | ✅ 代替 |
|---------|---------|
| 「スクリプトで対応します」 | 「対応進めていきます」 |
| 「ツールで自動化しました」 | 「修正しました」 |
| 「APIで取得した結果」 | 「確認したところ」 |
| 「自動生成しました」 | 「作成しました」 |

### 人名表記

チームメンバーの名前は **日本語表記で統一**。英語名・ローマ字名は禁止。
プロジェクトMEMORY.mdに各メンバーの正しい表記を記録しておく。

## 事前つつきコメント運用（★PJT-TD教訓）

定例前に、進捗確認・会話再開が必要なチケットにコメントを投げる運用。

### 毎週固定でつつくチケット

プロジェクトで「毎週必ず確認すべきチケット」をMEMORY.mdに定義:

```
@担当者
cc @他メンバー

進展あれば共有ください。
```

### 個別つつき

上記に加え、以下を追加:
- 会話が止まっているチケット
- 今週特に確認が必要なもの
- 期限が迫っているもの

### 運用ルール

- 過去の同様コメントを参照して文体を揃える
- **必ずユーザーに提示して確認を取ってから投稿**（定型コメントも含め全て確認必須）
- 出力フォーマット: テーブル（チケット名 / 宛先 / コメント案）で一覧化

→ Asana運用の詳細: [guides/11-asana-operations.md](11-asana-operations.md)

## 事実一覧（ファクトベース）運用（★PJT-YP教訓）

ヒアリングや会議で確認された**客観的事実**を蓄積・管理する仕組み。決定事項（DECISIONS.md）とは別軸で、方針策定や分析の根拠として活用する。

**特にシステム系案件で重要**: 業務フロー・既存システムの仕様・運用実態など、ヒアリングで得た事実を正確に記録しておくことで、誤った前提に基づく提案を防止できる。

### いつ使うか

- システム導入・刷新案件（業務フロー調査、現行システム仕様の把握が必要なケース）
- 複数ステークホルダーから異なる情報が出る可能性があるケース
- 長期案件でヒアリング内容の鮮度管理が必要なケース

※全案件で必須ではない。プロジェクト特性に応じて採用を判断する。

### 管理構造

- **FACTS.md** がSingle Source of Truth（マスタ）
- アジェンダシートに「事実一覧」シートを追加して同期可能（表示・共有用）
- 両者は同一内容を保持

### FACTS.md のフォーマット

```markdown
# 事実一覧（ファクトベース）

> DECISIONS.mdと同じ運用。確認済みの事実を蓄積し、プロジェクト方針・分析の根拠とする。
> アジェンダシートの「事実一覧」シートと同一内容を保持。

## [テーマ名]（例: 受注正常化）

| # | 事実 | 出典 | 確認日 |
|---|------|------|--------|
| F1 | [具体的な事実を記述] | [情報源: ○○ヒアリング / 課題管理表 等] | YYMMDD |
| F2 | ... | ... | ... |

## [テーマ2]

| # | 事実 | 出典 | 確認日 |
|---|------|------|--------|
| F20 | ... | ... | ... |
```

### 運用ルール

| 項目 | ルール |
|------|--------|
| **番号** | F1, F2... の通し番号。テーマ跨ぎで連番 |
| **事実 vs 決定** | 事実（F#）= ニュートラルな観察。決定（D#）= 合意された方針。混同しない |
| **更新タイミング** | ヒアリング直後・会議で確認事項が出た時点で追記（定例前後の定型アクションではない） |
| **確認日** | ファクトの鮮度管理。最終確認日を記録 |
| **テーマ分類** | 案件の主要論点ごとにセクション分け |

### Excel事実一覧シートの列構成（同期する場合）

| 列 | 内容 |
|----|------|
| A | # |
| B | 日付 |
| C | テーマ |
| D | 事実 |
| E | 出典 |
| F | 確認日 |

→ FACTS.md → Excel投入スクリプトで同期（adm-zip XML直接編集）
