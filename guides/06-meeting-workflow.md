# 定例運用フロー

> 定例会議の前後で実施するタスクの手順書。PJT-YPで確立したワークフロー。

---

## 定例前ワークフロー

定例前日または当日朝に実施。

### タスク1: スケジュール赤線の移動

1. `create-schedule-pptx.mjs` の `todayWeekFloat` を書き換え
2. `node create-schedule-pptx.mjs` を実行
3. 生成された `schedule.pptx` を確認
4. `drive/50.打ち合わせ関連/` に上書きコピー

**todayWeekFloat の計算方法**:
```
weekIndex + dayOffset/7
月曜=0, 火曜=1/7, 水曜=2/7, 木曜=3/7, 金曜=5/7
```

例: 金曜日の定例で第4週 → `todayWeekFloat = 4 + 5/7`

### タスク2: アジェンダシートの更新

1. スケジュールの `rawData` を参照し、今週の赤線が被るタスクを特定
2. Backlogで対象チケットの最新状況を確認
3. すり合わせが必要なトピックを選定
4. 前回アジェンダ行を `done` 化
5. 新規アジェンダ行を追加（adm-zip XML直接編集）
6. JSZipで検証

**アジェンダの構成**:
- **Row2（standing）**: スケジュール確認 — 今週のタスク一覧を記載
- **個別トピック行**: チケットごとに1行。提案型構造:
  - ▼前提（背景・制約）
  - ▼提案（FLUX-G提案内容）
  - ▼確認事項（合意したいポイント）

**並び順**: 戦略・ガバナンス系を先、技術・作業系を後

## 定例後ワークフロー

定例終了後に実施。

```
文字起こし受領
  │
  ▼
ステップ1: 抽出（チャットで一覧提示）
  ├─ 決定事項（番号付き）
  ├─ 議事メモ（議論の要点サマリ）
  └─ ネクスト（次のアクション一覧）
  │
  ▼
ステップ2: 記録（確認不要で即実行）
  ├─ DECISIONS.md 更新（通し番号を継続）
  ├─ Excel 決定事項シート更新（adm-zip）
  └─ docx 議事メモ更新（JSZip）
  → 完了報告 → ユーザー確認 → 修正あれば対応
  │
  ▼
ステップ3: ネクスト → フォーマットで一覧出力 → ユーザー確認
  → OKが出たらBacklog操作実行
```

### 文字起こし読み取りの注意事項

- **話者帰属は信頼しない**: マイク共有の場合あり。文脈から実際の話者を判断
- **日本の祝日は必ず自分で確認**: 文字起こしの「来週祝日」等を鵜呑みにしない
- **曖昧語禁止**: 具体的な数値・名称で記述

### DECISIONS.md更新のフォーマット

```
| # | 決定日 | 決定者 | 概要 | 詳細 | 備考 |
```

- 決定日: `YY/MM/DD` 形式
- 決定者: `合同` / `{会社名} {氏名}` 等
- 備考: 定例日付を記載（例: `260313定例`）

### ネクスト洗い出しフォーマット

| 項目 | 内容 |
|------|------|
| 内容 | やること概要 |
| 担当者 | Backlog上の担当者 |
| メンション | 通知する宛先 |
| 期限 | いつまでか |
| 紐づけ先 | 既存チケットの子課題 or 新規チケット |
| 対応方法 | 子課題作成 / コメント追記 / 新規チケット / 既存チケット更新 |

## 絶対厳守ルール

1. **客先に影響する操作は全て事前確認必須**: Backlogコメント投稿、担当者変更、ステータス変更
2. **チケット起票時**: カテゴリ・マイルストーンを必ず設定
3. **起票前に既存チケット検索**: 重複防止
4. **Backlogコメントはプレーンテキスト**: マークダウン記法禁止（AI感が出る）
