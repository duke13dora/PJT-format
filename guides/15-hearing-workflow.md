# ヒアリング資料の作り方

> 現場ヒアリング用のプレゼン資料を MD → すり合わせ → PPTX の3フェーズで作成するフロー。PJT-YPで確立した手法。

---

## 全体フロー

```
⓪ 情報収集
   ├─ 既存資料（業務フロー図、課題管理��、ヒアリング議事録等）
   ├─ Backlogチケット（関連チケットの概要欄・コメント）
   └─ 社内すり合わせ済みの方針MD
       │
       ▼
① ヒアリン��MDの作成
   └─ templates/hearing-format.md に沿ってPage単位で記述
       │
       ▼
② ユーザーとすり合わせ
   └─ MDベースで認識合わせ。質問の粒度・順序・表現を調整
       │
       ▼
③ フロー図の作成
   └─ Mermaid CLI / draw.io / PlantUML等でPNG生成
       │
       ▼
④ PPTX生成（FLUXフォーマット流し込み）
   └─ scripts/build-hearing-example.mjs をカスタマイズして実行
       │
       ▼
⑤ レビュー・ブラッシュアップ
   └─ 専門家ディスカッションで改善提案 → MD更新 → PPTX再生成
```

## 提案書との違い

| 観点 | 提案書 | ヒアリング資料 |
|------|--------|--------------|
| 目的 | 受注獲得 | 情報収集・認識合わせ |
| 読者 | 意思決定者 | 現場担当者 |
| トーン | 説得型 | 対話型（「確認したい」「教えてほしい」） |
| 構成 | 課題→提案→効果→費用 | 背景共有→現状フロー→あるべき姿→質問 |
| フロー図 | オプション | 必須（現状/あるべき姿の対比が核） |
| 質問 | なし | 各スライドに配置（メインコンテンツ） |

## フェーズ詳細

### ⓪ 情報収集

ヒアリング資料の質はインプット情報の量で決まる。以下を収集:

| 情報源 | 用途 |
|--------|------|
| 既存業務フロー図 | 現状フロースライドのベース |
| 課題���理表・課題構��� | 課題の全体像スライド |
| 他拠点との��較情報 | Appendix |
| 社内方針MD（あるべき姿・仕組み変更） | あるべき姿スライドのベース |
| 未回答の質問一覧 | 質問の網羅性チェック |

### ① ヒアリングMDの作成

`templates/hearing-format.md` をコピーして記述。Page単位の構造:

```markdown
## Page X: スライドタイ��ル
<!-- type: content | section: セクション名 -->

**タイトル**: スライドのタイトル

**リード文**: 1-2行の概要（現場の人が読んで「ああ、この話か」と分かるレベル）

**ボディ**:
- テーブル、フロー図参照、箇条書き等

**質問**:
- QXX: 質問文（質問ID付き）
```

#### 質問の書き方ガイドライン

| DO | DON'T |
|----|-------|
| 「〜は何割程度ですか？」（定量的に聞く） | 「〜についてどう思いますか？」（漠然） |
| 「〜の場合、A/B/Cのどれに近いですか？」（選択肢提示） | 「〜はありますか？」（Yes/Noで終わる） |
| 「〜の��度は？（毎日/週1/月1？）」（目安を添える） | 「〜の課題は？」（誘導的） |
| 質問IDを付ける（Q1, Q2...） | IDなし（後で追跡できない） |

### ② ユーザーとすり合わせ

MDベースで確認。特に以下を重点チェック:
- 質問の過不足（聞き漏れ/重複）
- 仮説の妥当性（「〜と想定」の部分）
- 先方が答えやすい構成になっているか

### ③ フロー図の作成

フロー図はヒアリング資料の核。以下の点に注意:

- **解像度**: ぼやけない方法で生成する（Mermaid CLIのデフォルト設定はぼやけがち）
- **一貫性**: 現状フローとあるべき姿フローで同じ記法・色使いを統一
- **凡例**: 色の意味（課題箇所=黄色、仕組み変更=青等）をフロー図内または近くに記載
- **仮説マーク**: 確認前のフローには「※仮��」を明記

ツール選択肢:
- **Mermaid CLI**: コードベースで管理しやすいが、日本語フォントとレンダリング品質に注意
- **draw.io**: GUI操作で見た目の調整が容易。SVG/PNG出力可
- **PowerPoint図形**: 最終的にPPTXに載せるなら直接作成も選択肢
- **PlantUML**: 複雑なフローに向くが、見た目のカスタマイズ性はMermaidより低い

### ④ PPTX生成

`scripts/build-hearing-example.mjs` をプロジェクト用にカスタマイズ:

1. スクリプトをプロジェクトディレクトリにコピー
2. スライド定義部分をMDの内容に合わせて編集
3. フロー図PNGのパスを設定
4. `node build-hearing-example.mjs` で生成

テン���レートは `templates/flux-presentation-template.pptx` を使用。
画像埋め込み時はテンプレー��の既存メディアファイル（image1-4.png等）と名前が衝突しないよう、`flow_1.png` 等のプレフィックス付き命名にすること。

### ⑤ ブラッシュアップ

叩き台を作成した後、以下の観点で専門家ディスカッション:
- 足りない要素・スライドはないか
- 質問の聞き方は適切か（オープンすぎ/誘導的すぎ）
- 構成順は現場の人が理解しやすい流れか
- 先方が答えやすい形式か（選択肢提示、具体例添え等）
- コンサルとして「こう見せた方が伝わる」等の提案

## スライド構成の設計原則

### 基本構成パターン

```
表紙 → アジェンダ → 概要(1-2枚) → 現状フロー(N枚) → あるべき姿概要 → あるべき姿フロー(N枚) → 過渡期 → 確認事項 → スケジュール → Appendix
```

### 設計原則

1. **現状→あるべき姿の対比**: 同じ業務パターン（例: 仕入品見積/在販見積/仕入品受注/在販受���）を現状・あるべき姿で対にする
2. **質問はフロースライドに��置**: 各フローの下に「▼確認したいこと」として質問を列挙
3. **横断テーマは専用スライド**: フロー固有でない確認事項（改善要望、過去の取り組み等）は「確認・ディスカッシ���ン事項」スライドにまとめる
4. **仮説は明示**: 確認前の情報は「※仮説ベース」と明記し、実態との差異を確認する姿勢を示す
5. **Appendixに他拠点比較**: パイロット拠点以外の情報は参考資料として末尾に

## 教訓・注意事項

### PPTX生成の教訓（PJT-YPより）

- **テンプ��ート画像との名前衝突**: `flux-presentation-template.pptx` には `ppt/media/image1-4.png`（ロゴ等）が存在。Mermaid等で生成した画像を `image1.png` 等の名前で追加するとテンプレート画像を上書きし、全スライドにゴースト画像が表示される。`flow_1.png` 等のプレフィックス付き命名で回避
- **adm-zip XML直接編集**: ExcelJS/SheetJS同様、PPTX編集もadm-zipでXML直接操作が安全
- **画像のrels**: スライドごとに `_rels/slideN.xml.rels` に画像リレーションを追加。他スライドのrelsには追加しない

### ヒアリング資料特有の注意

- **情報過多に注意**: 現場の人に見せる資料なので、1スライドの��報量は控えめに
- **「教えてほしい」スタンス**: 「こうすべき」ではなく「こう理解しているが合っているか」の姿勢
- **未確定情報は仮説と明記**: 先方に「決まったこと」と誤解されるリスクを避ける
