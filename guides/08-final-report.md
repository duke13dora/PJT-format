# 最終報告書の作り方

> プロジェクト完了時の最終報告書を MD → すり合わせ → PPTX の3フェーズで作成するフロー。PJT-TSで確立した手法。

---

## 全体フロー

```
① 最終報告書MDの作成
   └─ templates/final-report-format.md に沿って記述
       │
       ▼
② ユーザーとすり合わせ
   └─ MDベースで認識合わせ。中身の合意が取れてからPPTX化に進む
       │
       ▼
③ PPTX生成（FLUXフォーマット流し込み）
   └─ scripts/build-final-report-example.mjs をカスタマイズして実行
       │
       ▼
④ レビュー・微修正
   └─ PowerPointで開いて位置ずれ・重なりを手動修正
```

---

## フェーズ1: 最終報告書MDの作成

### 5章構成

| # | セクション | 目的 | ページ目安 |
|---|-----------|------|-----------|
| — | 表紙+アジェンダ | — | 2枚 |
| 01 | エグゼクティブサマリ | ここだけ読めば判断できる（結論・成果・提言を凝縮） | 扉+2-3枚 |
| 02 | プロジェクト概要 | 共通認識の確認（体制・スケジュール・スコープ） | 扉+2-3枚 |
| 03 | 実施内容と成果 | 各施策の内容・成果を適切な粒度で1回だけ説明 | 扉+8-12枚 |
| 04 | 課題と今後のアクション | 残課題・ブロッカー・Next Steps・提言（任意） | 扉+3-6枚 |
| 05 | 参考資料 | 必要な人だけ見る詳細（成果物一覧・決定事項等） | 必要に応じて |

### 設計原則

1. **ピラミッド原則**: 結論ファーストで構成。エグゼクティブサマリだけ読めば判断できる
2. **概要→詳細の2段構成は禁止**: 同じ内容を粒度を変えて2回説明しない。③で1回だけ適切な粒度で説明
3. **課題はメインセクション**: 「補足」に入れない。経営判断に直結する情報
4. **読者3層を意識**: 経営層（①のみ）/ マネージャー層（①+③+④）/ 実務担当者（全ページ）
5. **提言は④にマージ**: 経営判断を迫る場合のみ独立ページ化。報告型ならNext Stepsに含める

### MD記法

各ページは以下の形式で記述する:

```markdown
## Page N: [ページ名]

**タイトル**: [結論を言い切る形]

**リード文**: [1-2文。このスライドの「So What」]

**ボディ**:
[テーブル / 箇条書き / テキスト]
```

### ページ種別

| 種別 | 判定方法 | レイアウト |
|------|---------|-----------|
| 表紙 | Page 1 | slideLayout1 |
| アジェンダ | Page 2 | slideLayout7 |
| セクション扉 | ボディが「扉ページのため本文なし」 | slideLayout8（背景色+番号+タイトル） |
| コンテンツ | 上記以外 | slideLayout95（セクションラベル+タイトル+リード+ボディ） |

### コンテンツスライドの3層構造

```
┌──────────────────────────────────┐
│ [セクションラベル] 03 実施内容と成果 │  ← 小さい文字
│ [タイトル] PoC①は検証完了し…       │  ← 大きい文字（太字）
│ [リード] 7ヶ月間で3サイクルの…      │  ← 中サイズ黒文字
│──────────────────────────────────│
│ [ボディ]                           │  ← テーブル / テキスト / コールアウト
│  ┌──────────────────────────┐    │
│  │ テーブル or テキストボックス  │    │
│  └──────────────────────────┘    │
│  ┌──────────────────────────┐    │
│  │ コールアウトボックス（まとめ）│    │
│  └──────────────────────────┘    │
├──────────────────────────────────┤
│ ロゴ  © Corp. | CONFIDENTIAL    N │  ← フッター（テンプレート自動）
└──────────────────────────────────┘
```

### ④の内部構造

```
④ 課題と今後のアクション
  ├── 残課題一覧（区分ごとに分割）
  ├── ブロッカー
  ├── 今後のアクション（Next Steps）
  └── 提言（任意。経営判断を要する場合のみ記載）
```

### ユーザーとの認識合わせ

- MDの段階でユーザーに内容確認を依頼
- **中身の合意が取れてからPPTX化に進む**（PPTX化後の大幅な内容変更は手戻りが大きい）
- MD上でスライド枚数・各スライドの見出し・キーメッセージを確定させる
- 質問は1回のやり取りでまとめて投げる（逐次質問は負担が大きい）

### 適用パターン

- **報告型**（定期報告・完了報告）: 残課題+アクションだけ書く。提言は省略
- **提案型**（次フェーズ提案・追加投資判断）: 提言を厚めに書き、必要なら独立ページに

## フェーズ2: PPTX生成

### テンプレートPPTXの準備

- テンプレート: `templates/flux-presentation-template.pptx`（FLUX汎用PPTXテンプレート。5種のレイアウトサンプル付き）
- 保持する構成要素:
  - ロゴ（slideMaster内）
  - フッター
  - ヘッダーバー
  - スライドレイアウト（Layout1/7/8/95）
- **方針**: テンプレートに新規スライドを生成して追加

### 生成スクリプトの処理フロー

`scripts/build-final-report-example.mjs` をベースにカスタマイズ:

```
テンプレートPPTX (adm-zip読込)
    ↓
最終報告書.md をパース:
  - Page N の種別判定 (表紙/アジェンダ/扉/コンテンツ)
  - タイトル/リード/ボディの抽出
  - ボディ内のテーブル/テキストの分離
    ↓
各スライドのXMLを生成:
  - cover: buildCoverSlide()
  - agenda: buildAgendaSlide()
  - section: buildSectionSlide()
  - content: buildContentSlide() + makeTable/makeTextBox/makeCalloutBox
    ↓
presentation.xml / .rels / [Content_Types].xml 更新
    ↓
新PPTX出力
```

### XML構造の注意点（★重要）

| 項目 | 正しい方法 | NG / 原因 |
|------|---------|---------|
| テーブルセル | `<a:tc><a:txBody><a:bodyPr/><a:lstStyle/><a:p>...</a:p></a:txBody><a:tcPr>...</a:tcPr></a:tc>` | `<a:p>`を`<a:txBody>`で囲まないとセル内テキストが表示されない |
| rId（リレーションID） | rId100+i 等、既存rIdと衝突しない番号 | テンプレートのnotesMaster(rId26), fonts(rId27+)と衝突→スライド消失 |
| graphicFrame | `<p:graphicFrame>`でテーブルをラップ。`<p:xfrm>`で位置指定 | `<a:xfrm>`ではなく`<p:xfrm>`を使用 |
| 正規表現でのシェイプ操作 | callback-based replace | `/<p:sp>[\s\S]*?name="X"[\s\S]*?<\/p:sp>/` は前方のシェイプを巻き込む |
| notesSlide | スライドrelsからnotesSlide参照を除去 + 孤立ファイル削除 | 共有参照＋孤立ファイル→修復ダイアログ |

詳細: [09-pptx-xml-editing.md](09-pptx-xml-editing.md)

### レイアウト・位置ルール

| 項目 | ルール | 備考 |
|------|-------|------|
| スライドサイズ | 9144000 × 5143500 EMU（16:9） | 1inch = 914400 EMU |
| ボディ開始Y | y=1.5in以上 | リードテキスト下端の下 |
| フッター領域 | y=5.0in以下に収める | ロゴ+フッターテキストがy=5.0〜5.6にある |
| テーブル後の要素Y | テーブル下端 = y + (行数+1) × rowH | 後続要素はテーブル下端+0.1in以上 |
| フォントサイズ | sz="900"=9pt, sz="1000"=10pt | OOXMLは百分の一ポイント単位 |
| テキスト色 | 全て黒（000000） | 灰色（434343等）は禁止 |

### PPTXフォーマットルール

- **テーブル行番号**: 全テーブルに#列を必ず付ける。プレーン数字（1,2,3）を使用。丸数字（①②③）禁止
- **行の交互色**: ヘッダー=`{PRIMARY_COLOR}`白文字、奇数行=白、偶数行=薄色（例: F2F7FF）
- **行高さ統一**: 単一行は全テーブルで統一（0.28"）。複数行セルのみオーバーライド
- **リード文**: 12pt、黒色、文末に句点（。）必須
- **テーブル横幅**: リード文の幅と揃える（約9.2インチ）
- **フォント**: 全てメイリオで統一
- **敬称ルール**: 先方メンバーには「様」必須。自社メンバーには不要
- **ボディテキスト（テーブル外）**: 10pt（sz=1000）
- **アジェンダ**: テキスト形式で表示（カード/ボックス形式禁止）。セクション名は実際のセクション扉と一致させる
- **セクションラベル**: 太字
- **見出しラベル**: 太字、`{PRIMARY_COLOR}`色
- **担当者表記**: 残課題テーブルの担当は「{CLIENT_CODE}/担当者名」形式で統一。自社名は担当欄に記載しない（先方向け報告書のため）
- **残課題テーブル構造**: ステータス列は不要。列構成は `#, タスク, 担当, 備考`
- **コールアウトボックス**: 角丸四角形 roundRect adj=5000、背景色+テキスト色を指定

## フェーズ3: レビュー・修正

### 生成後チェックリスト

- [ ] PowerPointで修復ダイアログが出ないこと
- [ ] 全スライドが正しい順番で表示されること
- [ ] ロゴ、フッターが全スライドに表示
- [ ] セクションラベル（左上）が正しいセクション名
- [ ] タイトル/リード/ボディの3層が重ならず表示
- [ ] テーブルのセルにデータが入っていること
- [ ] コールアウトボックスがテーブルと重なっていないこと
- [ ] 最終スライドまで内容が入っていること
- [ ] 灰色テキストが残っていないこと

### よくあるフィードバックと対処法

| フィードバック | 対処 |
|-------------|------|
| テーブルの文字が見切れる | rowHを大きくする / fontSizeを小さくする |
| 要素が重なっている | 後続要素のy座標を下げる |
| フォントが違う | `typeface="メイリオ"`を確認 |
| 色が違う | srgbClr valを確認 |
| セクション扉が空 | buildSectionSlideのプレースホルダ置換ロジックを確認 |

### 重要原則

**XML直接編集の精度限界**: レイアウト微調整、フォントの再適用、コネクタの接続点ずれ等は完璧にできない。**最終的に人間がファイルを開いて目視確認・微修正する前提で作業すること。**
