# {PROJECT_NAME} — Claude Code設定

## コアルール

1. **言語**: ユーザーとのコミュニケーションは日本語
2. **進捗報告**: 5分以上沈黙禁止。長時間処理はステータス更新を送信。300秒超の待機は即停止し報告
3. **セキュリティ**: APIキー・パスワード・トークンのハードコード禁止。環境変数または `.mcp.json` の `env` で管理
4. **影響範囲確認**: 修正を加えた際は必ず影響範囲を確認し、関連箇所も修正すること
   - MCP設定変更 → 連携しているツール全てに影響
   - CLAUDE.md/MEMORY.md変更 → 全セッションに影響
   - 資料フォルダ構成変更 → 分析レポートの再生成が必要
   - **変更後チェックリスト**:
     1. この変更を参照/依存している箇所はどこか？
     2. それらの箇所は修正が必要か？
     3. ドキュメント（CLAUDE.md, MEMORY.md等）の更新は必要か？
5. **曖昧語禁止**: レポート・メッセージで曖昧な表現を使用しない。必ず具体的な数値・名称を記載
   - ❌「エラーが多発」→ ✅「15分間に3回発生」
   - ❌「一部で問題」→ ✅「8件中5件で障害」
   - ❌「概ね順調」→ ✅「5タスク中4タスク完了、1タスク待機中」
   - **適用範囲**: レポート、Slack通知、分析結果、MEMORY.md
6. **逆提案ルール**: 非効率な方法を指示された場合、より良い代替案を提案すること
7. **ファイル修正・再生成は明示的指示なしに禁止**:
   - 「差分を確認しろ」「認識しておいて」→ **確認結果の報告のみ**（ファイルに触らない）
   - 「修正しろ」「再生成しろ」→ 修正OK
   - 判断に迷う場合 → ユーザーに確認を取る
   - Office系ファイル（xlsx/docx/pptx）の再生成は特に注意（ユーザーが手動修正済みの可能性がある）
   - 背景: ユーザーが手動修正したファイルがClaude Codeに上書きされ、復旧不可能になったインシデントあり
8. **AI感排除**: 外部向けコミュニケーション（チケットコメント、メール等）でスクリプト・自動化ツールの存在を言及しない
   - ❌「スクリプトで対応します」→ ✅「対応進めていきます」
   - ❌「APIで取得した結果」→ ✅「確認したところ」
   - 人名は日本語表記で統一（ローマ字・英語名禁止）
   - 詳細: `guides/11-asana-operations.md` の「AI感排除原則」参照
9. **憶測回答禁止**: 確信がない情報は断定せず「確認が必要」と明示すること
   - ❌ 裏取りなしに「不要です」「問題ありません」と回答
   - ❌ チケット内容を確認せずに「対応済み」と回答
   - ✅ 「確認が必要です。〇〇を調べましょうか？」
   - ✅ 根拠（ファイル内容、チケット記載、DECISIONS.md）を示して回答
   - **適用範囲**: 全てのユーザーへの回答（特に先方に影響する判断）

## 3ストライクルール（詰まった時の対応）

同じ操作で**3回連続失敗**したら、場当たり的な試行錯誤を**即座に停止**。

### 検出時の対応手順

1. 即座に作業停止
2. EnterPlanModeでプランモードに入る
3. Exploreエージェントで調査
4. 根本原因分析（複数の専門家視点で）
5. 修正計画を策定してからリトライ

### 禁止事項

- 原因不明のまま何度もリトライ
- 「もう一度試してみます」で同じ失敗を繰り返す
- 根本原因を特定せずに次のステップに進む

## Office資料操作ガイド

Office系ファイル（Excel/Word/PowerPoint）をプログラムで操作する場合のルール。

### ツール選定（厳守）

| ファイル形式       | 操作         | 推奨ツール                         | 禁止ツール             | 理由                                     |
| ------------------ | ------------ | ---------------------------------- | ---------------------- | ---------------------------------------- |
| Excel (.xlsx)      | **書き込み** | adm-zip XML直接編集                | ExcelJS, xlsx(SheetJS) | 条件付き書式・フィルター・数式を破壊する |
| Excel (.xlsx)      | **読み取り** | exceljs（読み取り専用）またはJSZip | —                      | 読み取り専用なら安全                     |
| Word (.docx)       | **編集**     | JSZipでXML直接編集                 | —                      | レイアウト・スタイル保持のため           |
| PowerPoint (.pptx) | **編集**     | JSZipでXML直接編集                 | —                      | 同上                                     |

### フォーマット厳守原則

既存のファイルに追記・編集する場合、**フォント、罫線、インデント、スタイルは既存エントリと完全一致させる**こと。XML直接編集なら既存要素をコピーして値だけ変更するアプローチが安全。

### 典型的な実装パターン

```javascript
// Excel書き込み（adm-zip XML直接編集）
import AdmZip from 'adm-zip';
const zip = new AdmZip('input.xlsx');
const sheetXml = zip.readAsText('xl/worksheets/sheet1.xml');
// XMLを解析して行追加
zip.updateFile('xl/worksheets/sheet1.xml', Buffer.from(updatedXml));
zip.writeZip('output.xlsx');

// Word編集（JSZip XML直接編集）
import JSZip from 'jszip';
const zip = await JSZip.loadAsync(fs.readFileSync('input.docx'));
const docXml = await zip.file('word/document.xml').async('string');
// XMLを解析して段落追加
zip.file('word/document.xml', updatedXml);
const buf = await zip.generateAsync({ type: 'nodebuffer' });
fs.writeFileSync('output.docx', buf);
```

### XML直接編集の精度限界

XML直接編集で構造的な変更は概ね正確に行えるが、以下は完璧にできない:
- レイアウト微調整（座標のピクセル単位のズレ、余白、文字切れ）
- フォントの再適用（XMLで指定しても反映されないケースあり）
- コネクタの接続点ずれ

**最終的に人間がファイルを開いて目視確認・微修正する前提で作業すること。**

### OOXML正規表現の致命的バグパターン

`/<p:sp>[\s\S]*?name="X"[\s\S]*?<\/p:sp>/` は**使用禁止**。最初の`<p:sp>`からターゲットまで全シェイプを巻き込んで削除する。**callback-based replace**を使うこと:
```javascript
xml = xml.replace(/<p:sp>([\s\S]*?)<\/p:sp>/g, (match, content) => {
  if (content.includes('name="TargetName"')) return '';
  return match;
});
```

### バックアップ運用

Office系ファイルを編集する前に、必ずバックアップを取ること:
- **バックアップ先**: `{PJ_ROOT}\backup\` フォルダ
- **命名**: `元ファイル名_backup_YYMMDD.xlsx`（例: `アジェンダ_backup_260219.xlsx`）
- `_tmp` ファイルも作業完了後に `backup/` へ移動
